# 原則

## 根公理

**媒合的正當性 = 適配性決定資格，公平程序決定結果。**

適配性（屬性、規則）不直接決定誰拿到，它只決定「誰有資格被抽」；
最終由可驗證的隨機程序決定誰真的拿到。
最佳化只發生在資格門檻內，**不在資格集合外做最佳化**。

## 衍生原則

### 1. 屬性與規則必須可解釋

- **推導自**：根公理（要定義「資格」就必須能說清為什麼有/沒資格）
- **意義**：所有用於媒合的屬性（老師專業、班級特色）與規則（資格條件），
  必須是顯式、人類可讀的資料；不可使用黑箱分數作為硬性過濾條件。
- **如何應用**：規則以可讀格式（如 YAML/JSON/表格）儲存；
  每條規則都附帶自然語言說明，讓學校行政能直接看懂。
  「模板」是規則可解釋性的可重用容器——把屬性 schema 與規則打包成可命名、
  可分享的單元（見 `vision.md` 架構：模板系統）。

### 2. 分配過程必須可驗證且可重現

- **推導自**：根公理（程序公平 = 第三方能驗證結果未被動手腳）
- **意義**：分配機制（純抽籤、志願序分配等）所引入的隨機性，必須源自可公開的種子
  （例：公開彩券號碼、會議現場擲骰），且整個分配過程能用相同種子重新跑出相同結果。
- **如何應用**：分配模組必須接受外部 seed 輸入；輸出完整的分配過程紀錄
  （包含每一步的選擇與隨機決策），任何人拿到（規則 + 名單 + 志願 + seed）
  都能複現結果。

### 3. 不在資格集合外做最佳化

- **推導自**：根公理（一旦在資格外做最佳化，就等於用適配性蓋過公平）
- **意義**：先以規則篩出「可行配對集合」，再在集合內以公平的分配機制決定結果；
  純抽籤與志願序皆為分配機制的具體實現，禁止在分配過程內偷塞分數加權或偏好權重
  （志願是當事人自己的選擇，不算系統強加的最佳化）。
- **如何應用**：演算法必須能輸出「資格集合」這個中間產物作為可稽核快照；
  程式碼結構上應將「過濾」與「分配」明確分離為兩個階段。

### 4. 結果可稽核

- **推導自**：原則 1 + 原則 2
- **意義**：對任何一個配對結果，能回答兩個問題——
  「為什麼這個組合落在資格集合內？」與「抽籤過程怎麼得出它？」
- **如何應用**：每次媒合都保留：規則快照、輸入名單、資格集合、
  隨機種子、抽籤步驟紀錄。學校行政可匯出為稽核報告。

### 5. 對使用者透明

- **推導自**：原則 1 + 原則 4
- **意義**：被媒合的當事人（老師、班級代表）應能查到自己為什麼有/沒資格、
  是在哪一輪被抽到或沒抽到。透明是化解爭議的基礎。
- **如何應用**：提供「個別查詢」視圖；用語面向學校行政與一般教師，
  避免技術名詞。
