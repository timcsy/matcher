# Feature Specification: 核心媒合引擎（library + CLI）

**Feature Branch**: `001-core-allocator`
**Created**: 2026-05-22
**Status**: Draft
**Input**: User description: "核心媒合引擎：純函式 library + CLI，依規則過濾出資格集合，在 M0 純抽籤分支下完成分配，輸出可重現的稽核紀錄。介面預留 preferences 參數，但本階段僅實作 preferences 為空時的純抽籤分支。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 — 完成一次可重現的純抽籤媒合（Priority: P1）

學校行政在 CLI 上提供一份「規則檔」（描述哪些角色有資格被分到哪些對象）、一份「名單檔」（所有待媒合的角色與對象，含對象容量）、以及一個「隨機種子」。系統先輸出「資格集合」中間結果，接著在資格集合內以純抽籤決定最終配對，並產出一份完整稽核紀錄。

**Why this priority**：這是 matcher 根公理「適配性決定資格、公平程序決定結果」的最小完整實作；沒有它，後續模板、Web UI、志願序模式皆無從談起。

**Independent Test**：在 CLI 上以基準場景「教師-班級配對」執行一次媒合，比對輸出的稽核紀錄、資格集合與最終配對是否齊全且互相吻合；此測試本身就是學校行政會做的事，可作為 MVP demo。

**Acceptance Scenarios**：

1. **Given** 一份規則檔（教師專業需在班級需要科目列表中）、一份名單檔（10 位老師、5 個班級、每班容量 2）、一個任意 seed，**When** 執行媒合，**Then** 系統依序輸出：(a) 資格集合（哪些老師對哪些班級「有資格」），(b) 最終配對結果（每位老師被分到的班級），(c) 完整稽核紀錄。
2. **Given** 與情境 1 完全相同的輸入（規則、名單、seed），**When** 再次執行，**Then** 兩次得到 100% 一致的資格集合、最終配對與稽核紀錄。
3. **Given** 一份不同的 seed 但其餘輸入相同，**When** 執行媒合，**Then** 資格集合保持一致（規則篩選結果不變），但最終配對結果可能不同（抽籤的合法多樣性）。

---

### User Story 2 — 邊界情境的明確錯誤回應（Priority: P2）

當輸入無法產生有效配對時（例如「沒有任何老師符合任何班級的資格條件」、「資格集合內可分配名額不足以滿足所有角色」、「規則互相矛盾」），系統 MUST 以明確的錯誤訊息回應，並指出問題類型與位置，**不可**靜默回傳空結果或預設值。

**Why this priority**：學校行政若拿到「靜默成功」的空結果，可能誤以為媒合完成；明確錯誤是公平與可稽核性的最後防線。

**Independent Test**：構造三種異常輸入（全員無資格、容量不足、規則互斥），執行後檢驗錯誤訊息是否清楚指出原因。

**Acceptance Scenarios**：

1. **Given** 一份「沒有任何老師專業符合任何班級需求」的輸入，**When** 執行媒合，**Then** 系統回應「資格集合為空」的明確錯誤，並指出至少一位老師與一個班級之間的不匹配原因。
2. **Given** 一份「老師人數 > 所有班級總容量」的輸入，**When** 執行媒合，**Then** 系統回應「容量不足以容納所有角色」並列出超額人數。
3. **Given** 一份規則檔內部互相矛盾的輸入（例如同一條規則同時要求 A 與非 A），**When** 執行媒合，**Then** 系統在過濾階段即回應規則互斥錯誤，不進入分配階段。

---

### User Story 3 — 為未來機制保留介面（Priority: P3）

核心引擎的介面 MUST 接受可選的「志願表」(preferences) 參數，但在本階段（僅支援 M0 純抽籤），若使用者提供非空 preferences，系統 MUST 明確拒絕並提示該機制不接受志願輸入，並指引使用者：志願序機制（M1/M2）將於後續階段（階段 4）加入。

**Why this priority**：這項保證了未來擴充志願序模式時，呼叫端介面不需重構；同時避免使用者誤以為現階段已支援志願。

**Independent Test**：在 CLI 上提供帶有 preferences 的輸入，驗證系統明確拒絕並輸出對應提示，未進入分配階段。

**Acceptance Scenarios**：

1. **Given** 一份附有 preferences 欄位（非空）的輸入，**When** 執行媒合（機制為 M0），**Then** 系統拒絕執行並回應「此機制（M0 純抽籤）不接受志願輸入；若需志願序，請等待 M1/M2 機制」。
2. **Given** 一份 preferences 欄位明確為空或不存在的輸入，**When** 執行媒合，**Then** 系統正常進入純抽籤流程。

---

### Edge Cases

- **空名單**：輸入名單為空 → 系統回應「無待媒合角色」明確錯誤。
- **seed 未提供**：未指定 seed → 系統拒絕執行，不可使用系統時鐘或內部 RNG 作為預設值。
- **重複身分**：同一位老師在名單中出現多次 → 系統回應「名單有重複身分」錯誤並指出位置。
- **未知屬性**：規則引用了名單中不存在的屬性 → 系統在過濾前回應「規則引用未定義屬性」錯誤。
- **稽核紀錄不可重現**：若任何隨機決策未被紀錄足以重播 → 視為缺陷（本規格的核心反例）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**：系統 MUST 接受人類可讀的結構化規則檔（格式由實作階段決定，但須可被非技術人員肉眼讀懂）。
- **FR-002**：規則檔的每條規則 MUST 附帶自然語言說明欄位，使學校行政能直接看懂該規則在判斷什麼。
- **FR-003**：系統 MUST 接受外部提供的整數種子（seed）作為唯一隨機性來源；MUST 拒絕在 seed 未提供時執行。
- **FR-004**：系統 MUST 在「分配」前先輸出「資格集合」作為獨立可檢視的中間產物（即使分配後續失敗，資格集合仍可單獨輸出）。
- **FR-005**：系統 MUST 在程式碼/介面上將「過濾」與「分配」明確分離為兩個階段；外部呼叫者可只執行「過濾」而不執行「分配」。
- **FR-006**：系統 MUST 以 M0 純抽籤機制將資格集合內的角色分配給對象，在對象容量約束內進行。
- **FR-007**：系統 MUST 在輸入相同（規則 + 名單 + seed + preferences）時，100% 重現相同的資格集合、最終配對與稽核紀錄。
- **FR-008**：系統 MUST 輸出完整稽核紀錄，至少包含：（a）規則檔快照，（b）名單檔快照，（c）資格集合，（d）使用的 seed，（e）所選機制名稱（本階段固定為 M0），（f）分配過程的每一步隨機決策。
- **FR-009**：稽核紀錄 MUST 足夠完整，第三方僅憑紀錄即可獨立驗算（不需要重跑程式即可閱讀「為什麼這個配對在資格集合內」與「抽籤如何得出最終配對」）。
- **FR-010**：系統介面 MUST 包含可選的「志願表」(preferences) 參數；當機制為 M0 時，若 preferences 非空，系統 MUST 明確拒絕並提示志願序機制將於後續階段加入。
- **FR-011**：系統 MUST 在以下情境輸出明確錯誤（不可靜默回傳預設值）：資格集合為空、容量不足、規則互斥、名單為空、名單有重複身分、規則引用未定義屬性、seed 未提供。
- **FR-012**：系統 MUST 提供「教師-班級配對」基準場景的範例規則檔與名單檔，並能用 CLI 跑通該範例。
- **FR-013**：規則的表達能力 MUST 至少支援屬性等值匹配、邏輯運算（AND / OR / NOT）、以及範圍/集合運算（≥、≤、IN）；不需支援自訂 DSL 或外部函式呼叫。
- **FR-014**：所有錯誤訊息、提示與稽核紀錄中的可閱讀說明文字 MUST 為繁體中文（依 constitution 的繁體中文文件原則）。

### Key Entities

- **角色（Role）**：要被分配的個體，具有可命名的屬性集合（例如老師具有「專業」、「年資」、「姓名」）。每個角色在一次媒合中至多被分配到一個對象。
- **對象（Target）**：被分配的容器，具有屬性集合與一個容量上限（capacity ≥ 1）。例如班級具有「需要科目」、「特色」、「容量」。
- **規則（Rule）**：定義「哪些角色屬性 vs. 哪些對象屬性符合資格」的可解釋判斷式，附自然語言說明欄位。
- **資格集合（Qualified Set）**：通過規則篩選的所有合法角色-對象 候選配對的集合；是過濾階段的最終輸出與分配階段的唯一輸入。
- **配對結果（Assignment）**：每個角色實際被分配到的對象（在容量限制內）。
- **稽核紀錄（Audit Record）**：可重現整個流程的完整紀錄（規則快照、名單快照、資格集合、seed、機制名稱、每步隨機決策、最終配對）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**：給定相同輸入（規則 + 名單 + seed + preferences）執行兩次，兩次的稽核紀錄逐位元組相同（100% 一致）。
- **SC-002**：「教師-班級配對」基準場景（10 位老師、5 個班級、每班容量 2）能在 CLI 上於 5 秒內完成並產出稽核紀錄。
- **SC-003**：七種預定義邊界情境（資格集合為空、容量不足、規則互斥、名單為空、名單有重複身分、規則引用未定義屬性、seed 未提供）100% 產生明確錯誤訊息；0% 靜默通過。
- **SC-004**：seed 未提供時，100% 拒絕執行；不存在「無 seed 仍能跑出結果」的路徑。
- **SC-005**：稽核紀錄的可讀性測試：找一位未參與開發的學校行政，僅憑稽核紀錄能在 10 分鐘內回答「為什麼老師 X 被分到班級 Y」與「為什麼老師 X 沒被分到班級 Z」這兩個問題。
- **SC-006**：在 M0 機制下提供非空 preferences 時，100% 收到明確拒絕訊息並指引未來階段；0% 靜默忽略。
- **SC-007**：基準場景的核心測試覆蓋：規則篩選、邊界情境、可重現性、preferences 拒絕邏輯——四項皆有自動化測試（依 constitution 的 TDD 原則）。

## Assumptions

- **基準場景的配對形狀**：採「每位老師恰好分到 1 個班；每個班可收多位老師（容量 ≥ 1）」（many-to-one with target capacity）。多對多形狀屬於 scheduler 範疇，已在 vision.md「範圍邊界」明文排除。
- **介面形態**：純函式 library + CLI 包裝；CLI 主要用於本階段的驗證與 demo，不涉及 Web UI（屬階段 3）。
- **規則/名單檔格式**：採人類可讀的結構化檔案（YAML 或 JSON 由 `/speckit.plan` 決定）；本階段不涉及 CSV/Excel 匯入（屬階段 2）。
- **稽核紀錄格式**：採結構化檔案（JSON 為合理預設），同時具備可機讀（重播驗算）與可肉眼閱讀（紀錄中含自然語言說明）兩種屬性。
- **模板系統**：本階段不抽象模板；「教師-班級配對」schema 直接寫死於範例檔，模板系統屬階段 2。
- **隨機性實作**：選定的 RNG MUST 是 seedable 且跨平台/跨語言版本結果一致（具體選型於 `/speckit.plan` 決定）。
- **使用者**：本階段無 UI、無認證；CLI 操作者為開發者或行政測試人員。
- **不處理**：Web UI、CSV/Excel 匯入、模板系統、志願序機制（M1/M2）、加權抽籤、先到先得、雙邊穩定匹配（DA）、排程、投票——皆於 vision.md 路線圖或範圍邊界中明確指派他階段或排除。
